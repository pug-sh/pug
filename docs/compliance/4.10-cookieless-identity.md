## 1. What this is, and what it is *not*

A **cookieless identity mode**: for a visitor with no client-stored identifier, the *server* derives a synthetic, daily-rotating anonymous ID by hashing request attributes it already sees. This is the model OpenPanel, Rybbit, and PostHog's `cookieless_mode` all use. It lets a controller count unique visitors **without writing anything to the visitor's device and without the identifier being personal data** â€” the basis for measuring consent-denied visitors (and, for some controllers, skipping the banner entirely).

It is **not** a replacement for Pug's default identity model. Pug is positioned as **product analytics**, not privacy-first web analytics. The persistent client UUID (`anon-<uuid>` in `profile.ts`, session ID in `session.ts`) stays the **default**, because anonymous cross-day retention and the anonymousâ†’identified server-merge are core product-analytics features that a server-hash-only model structurally breaks (the hash rotates daily, so there is no stable anon ID to retain or to merge a pre-signup history onto). See the positioning decision: keep UUID default, add this as opt-in.

**The chosen shape is PostHog `on_reject`, not `always`:**

| Consent state | Identity used | Client storage | `identify()` |
| --- | --- | --- | --- |
| **granted** | client UUID (`anon-<uuid>`) â†’ `external_id` after `identify()` | localStorage (today's behaviour) | allowed |
| **denied / pre-consent** | **server-hash** `H` (this doc) | none | blocked |

On `optInTracking()` the SDK stops sending the cookieless flag and switches to the normal UUID profile â€” the visitor *upgrades* from anonymous-counted to fully-tracked. There is intentionally no attempt to stitch the pre-consent server-hash identity onto the post-consent UUID (doing so would re-link the very data the visitor declined â€” privacy-wrong and pointless).

## 2. Server-side mechanics

### 2.1 The hash

```
H = hash( project_id, daily_salt, client_ip, user_agent, host )
```

- Reuses the **transient** client IP already extracted for geo (`geo.ClientIP` from `CF-Connecting-IP`, `internal/geo/cloudflare.go`) â€” the IP is hashed and discarded in the same request, never stored (consistent with Â§4.4, which already strips `$ip` everywhere).
- `host` (event `$url` origin) namespaces per-site so the same person on two customer sites doesn't collide.
- Output replaces `distinct_id` and seeds `session_id` for these events.

### 2.2 Salt lifecycle â€” the whole legal premise

The salt is what makes `H` **anonymous rather than pseudonymous**: rotate it daily and destroy the old value, and yesterday's hashes can no longer be re-derived or linked to a person. Discipline required:

- One random salt per **UTC day** (global, or per-project if we want per-tenant unlinkability). Generated lazily, held in memory/Redis.
- **Never persisted to durable storage, never logged, never put in a trace** (cf. Â§4.8 telemetry-scrubbing â€” a salt in SigNoz would defeat the entire design).
- TTL ~**48h**, not 24h: keep **today's + yesterday's** salt live so an event arriving just after midnight (or a slightly-skewed client clock) still resolves. At ingestion, compute against today's salt; on a session-continuity miss at the boundary, fall back to yesterday's. (This mirrors OpenPanel/Rybbit's "current + previous day" handling.)

### 2.3 Strict by default

Rybbit exposes a salt **on/off** toggle and defaults to **off** (a *stable* cross-day hash) â€” which their own docs concede "may be considered storing personally identifiable data under GDPR." We should default to the **strict, daily-rotating** posture (salt **on**), because the entire reason to offer this mode is the no-personal-data guarantee. A stable-hash variant, if ever offered, is a separate opt-in with its own banner story â€” not the default.

### 2.4 Profiles, erasure, merge

- These profiles are **synthetic/anonymous** â€” flag them as such in the profiles model (`docs/architecture/profiles.md`) so the alias-resolution machinery from Â§4.1 treats them correctly.
- They are inherently **outside DSAR scope**: with no external linkage and a destroyed salt, there is no way to map a person to their hashes â€” erasure/export by `external_id` simply has nothing to resolve. This is consistent, not a gap (the data is already anonymous).
- They never participate in the anonâ†’identified **merge** (no stable anon ID exists to merge).

## 3. SDK changes (`../../../sdk-web`)

Small, and largely reuses code that already exists:

- A `cookieless` opt-in (likely folded into an `InitOptions.persistence: 'memory'` / `cookieless` flag). While consent is **denied**, the SDK takes the existing storage-blocked in-memory path (`fallbackSessionId` in `session.ts`, in-memory anon in `profile.ts`) **by choice**, writes **no** localStorage, and sends events with empty `distinctId`/`sessionId` plus a "derive server-side" marker.
- `identify()` is **blocked** while in the server-hash state (a distinct ID would be personal data â€” the cookieless premise forbids it).
- On `optInTracking()`: drop the marker, switch to the normal UUID profile (the upgrade). This rides on the consent gate in `src/tracking-consent.ts` that already reconciles listeners on opt-in.

The ingestion handler (`internal/app/server/rpc/sdk/events/handler.go`) must **trust the marker only to derive `H`** â€” it must never let a client smuggle in its own `distinct_id` alongside the cookieless flag (same untrusted-input hardening already applied to `$ip` in Â§4.4).

## 4. What the controller gains / loses

**Gains:** counts 100% of visitors including consent-denied ones; for controllers whose policy permits, a banner-free analytics deployment; no `localStorage` writes for the anonymous population (sidesteps the ePrivacy "terminal equipment" trigger noted in Â§4.7).

**Loses (vs. the UUID default â€” hence opt-in):** anonymous cross-day retention; the anonâ†’identified pre-signup merge; precise uniqueness (server-hash collides users behind one NAT IP + UA and splits a user who changes IP). These are exactly the product-analytics capabilities we are unwilling to give up *by default* â€” which is why this is a mode, not the model.

## 5. Open questions

- **Salt scope:** global per-day vs. per-project. Per-project gives cross-tenant unlinkability but multiplies the in-memory salt set; global is simpler and `project_id` is already a hash input. Lean global unless a customer needs per-tenant unlinkability.
- **Salt store:** in-process memory (simple, but each replica holds its own salt â†’ same visitor hashes differently across replicas, splitting counts) vs. shared Redis (consistent across replicas; the salt now lives in Redis and must still be TTL'd and kept out of backups). The replica-consistency issue likely forces Redis.
- **Sessionization** for cookieless events: derive `session_id` from `H` + a coarse time bucket, or run a server-side sessionizer. Needs a short spike.
- **Do we even surface salt-off (stable hash)?** Recommendation: no, not initially â€” it's the legally-muddiest mode and not why we're building this.

## 6. Done when

- A project flagged cookieless ingests events with **no** client identifier; the server populates `distinct_id`/`session_id` from `H`.
- The daily salt provably rotates, is held â‰¤48h, and appears in **no** durable store, log, or trace.
- Consent-denied visitors are counted; on opt-in, the same browser switches to the UUID profile with no attempt to stitch the prior hash.
- `identify()` is rejected while cookieless; these profiles are flagged synthetic and excluded from merge and from DSAR alias resolution.
