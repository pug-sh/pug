# 4.1 — Erasure that reaches the events: implementation scope

Scopes **§4.1** of [`roadmap.md`](./roadmap.md) — the headline P0 failure: `ProfilesService.Delete`
soft-deletes the profile but never touches the `events` table, so a data subject's behavioral
history survives forever (fails GDPR Art. 17 / DPDP §12).

**Status:** implemented.
**Context:** pre-launch — **no backward-compatibility constraints** (proto/schema may change freely).
Build for the stricter GDPR/DPDP superset.

---

## Decisions locked during scoping

1. **`dashboard_event_rollup_daily` is left in place** as an out-of-scope anonymous aggregate
   (decision "a" — see [§5](#5-the-materialized-view-problem)).
2. **Migrations fold in place** — existing tables are edited in their original migration files, not
   ALTER-ed via new ones.
3. **The DSAR audit trail ships as a unified `compliance_requests` table**
   (`015_create_compliance_requests.sql`), discriminated by `kind` (`'erase'` | `'export'`) so
   erasure (§4.1) and export (§4.2) share one ledger. It is purely additive (modifies no existing
   table), so it does not conflict with the no-ALTER rule above. *(Originally scoped as a delete-only
   `deletion_requests` table; unified when §4.2 was scoped — pre-launch, rewritten in place.)*
4. **The `events` sort key is NOT changed** — see [§5](#5-the-materialized-view-problem).
5. **Mutation wait = strategy A** (synchronous, generous timeout) — see [§7](#7-clickhouse-mutation-strategy).
6. **Erasure runs in a generalized `compliance` worker**, not a dedicated erase worker. It's the home
   for the slow, low-volume DSAR/governance jobs that share a long-timeout consumer profile — 4.1
   erasure now, with 4.2 export and 4.5 retention/TTL adding sibling consumers in the same process.
   (A separate *consumer* is required regardless of volume: a 5-minute `mutations_sync` job can't share
   the hot-path workers' 30-second ack window.)

---

## 1. Goal & acceptance

**Done when:** after an erasure request completes —
- the subject's profile **and every event row across all their `distinct_id`s** are gone from ClickHouse;
- the per-person derived rollups (`distinct_id_activity_states`, `dashboard_session_rollup`) are gone;
- the PostgreSQL `profiles` + `profile_devices` rows are gone;
- the `compliance_requests` row (kind `erase`) shows `completed` with a completion timestamp and counts;
- a re-query for the subject's **raw events and profile** returns nothing (the
  anonymous `dashboard_event_rollup_daily` aggregate is retained — see §9).

`dashboard_event_rollup_daily` is **intentionally retained** as an anonymous aggregate
(documented limitation, [§9](#9-known-limitations-deferred)).

---

## 2. Identity model: what to erase

A person's events are keyed by `events.distinct_id`, which stores the **raw string the SDK sent**,
unchanged. Over a lifetime that string is *all* of:

- one or more `anon-<uuid>` values (pre-identify), recorded as **alias rows**;
- the customer-supplied **`external_id`** (the SDK sends `distinct_id = external_id` after `identify()`);
- (for completeness) the internal profile `id`, which the read-path activity CTE also treats as a
  possible `distinct_id` (`profileActivitySummaryCTE` in `internal/core/profiles/service.go`).

`profile_aliases` (ClickHouse-only — there is **no** PostgreSQL alias table) holds the anon→profile
history. The complete erase key set is:

```
distinct_id_keys = {external_id}                       -- always, the literal request key
                 ∪ {profile.id}                         -- iff a profile resolves
                 ∪ {alias_id … FROM profile_aliases FINAL
                       WHERE project_id = ? AND profile_id = ?}   -- iff a profile resolves

session_ids      = SELECT DISTINCT session_id FROM events
                     WHERE project_id = ? AND distinct_id IN (distinct_id_keys)
```

### `external_id` is a first-class erase key, not just a lookup handle

Because `events.distinct_id` can equal the raw `external_id`, events keyed by `external_id` exist
**independently of, and can outlive, the profile row.** Therefore:

- `external_id` is always in `distinct_id_keys` (above), so the events `DELETE` catches those rows.
- **Erasure by `external_id` must proceed even when no profile resolves.** The request is created and
  the worker erases everything keyed on the `external_id` string. The only thing unavailable without
  a profile is the *anon* id history (nothing links those anon ids to the person) — an inherent,
  accepted limit.

`external_id` is also the **primary** controller-facing handle: a "delete me" flow knows the user by
the id the controller assigned, not by pug's internal profile xid.

### Derived anonymous persons are erasable by id

Profiles reads surface **derived anonymous persons** — unclaimed `distinct_id`s from
`distinct_id_activity_states` with no profile row anywhere (see
[`../architecture/profiles.md`](../architecture/profiles.md)). For these, **the person id IS the
events `distinct_id`**, and `Delete` (by id) must accept it. `RequestErasureByID` resolves a
Postgres-missing id through a ladder, in order:

1. **alias** claimed by an identify merge → erase the **canonical** profile (its fan-out covers the alias);
2. **external_id-shaped id** (the user key an events row displays post-identify) → route to the
   `external_id` path;
3. **derived anonymous person** (has activity rollup rows) → erase by the bare `distinct_id`:
   the ledger row records it as `profile_id`, `distinct_id_keys = {id}`, and the worker's Postgres
   deletes are rows-affected no-ops;
4. none of the above → `ErrProfileNotFound`.

Because ledger `profile_id` now carries arbitrary-length distinct_ids, the column is `text`
(migration `016_compliance_profile_id_text.sql` — `char(20)` also space-padded shorter ids, which
would corrupt the frozen fan-out).

## 3. API surface

All erasure RPCs live on the **shared** service (`proto/shared/profiles/v1/profiles.proto`) — **private
API key or JWT**, never the public SDK.

> **Why not the SDK surface** (despite the roadmap's "shared/SDK" phrasing): public keys are
> extractable from client apps, and an untrusted caller able to erase arbitrary `external_id`s is a
> deletion-DoS. Erasure must be driven by the controller's *backend* after it verifies the end-user.

```proto
service ProfilesService {
  // ... existing Get / GetByExternalId / List ...
  rpc Delete(DeleteRequest) returns (DeleteResponse) {}                          // by profile id
  rpc DeleteDataSubject(DeleteDataSubjectRequest) returns (DeleteDataSubjectResponse) {}  // by external_id
  rpc GetDeletionRequest(GetDeletionRequestRequest) returns (GetDeletionRequestResponse) {} // audit / proof
}
```

- `Delete` and `DeleteDataSubject` are two front doors to **one** erasure pipeline. `DeleteResponse`
  changes to return `{ request_id, status }` (free, pre-launch).
- `GetDeletionRequest` returns status + timestamps + counts so a controller can **prove** fulfilment
  within the statutory window.
- Worker message (`proto/workers/compliance/v1/compliance.proto`):
  `EraseMessage { project_id, request_id, profile_id (optional), external_id }`.

## 4. Schema changes

### Unified DSAR ledger: `compliance_requests` (PostgreSQL)

`015_create_compliance_requests.sql` (see decision 3) — one audit table for all data-subject-rights
requests, discriminated by `kind`. Erasure (§4.1) writes `kind = 'erase'`; export (§4.2) writes
`kind = 'export'`:

```sql
-- +goose Up
create table compliance_requests (
  id              char(20)    primary key,
  project_id      char(20)    not null references projects(id) on delete cascade,
  kind            text        not null check (kind in ('erase', 'export')),
  profile_id      text,              -- resolved subject (xid, or the distinct_id for a derived anonymous
                                     -- person — arbitrary length, hence text via migration 016);
                                     -- NO fk: profile is hard-deleted, audit row must survive
  external_id     text,              -- what was requested (audit: "what")
  status          text        not null default 'pending'
                    check (status in ('pending','processing','completed','failed')),
  distinct_ids    text[],            -- erase-only: frozen fan-out (retry-safe + proof of what was erased); NULL for export
  session_ids     text[],            -- erase-only: frozen session set (retry-safe); NULL for export
  events_affected bigint      not null default 0,   -- erased (erase) or exported (export), by kind
  requested_by    char(20),          -- accountability: customer id (JWT); null for API-key callers
  requested_at    timestamptz not null default now(),
  completed_at    timestamptz,
  update_time     timestamptz not null default now(),
  error           text
);
create index compliance_requests_project_kind_status on compliance_requests (project_id, kind, status);
create trigger update_timestamp before update on compliance_requests
  for each row execute procedure moddatetime(update_time);

-- +goose Down
drop table compliance_requests;
```

`profile_id` is deliberately **not** a foreign key: the profile is hard-deleted, and the audit row
must outlive it. The erase-only `distinct_ids`/`session_ids` are NULL for export rows.

### No ClickHouse migration change

Erasure is **hard delete** (`ALTER … DELETE`), so the `events` table needs no `is_deleted` column and
no TTL. The sort key is unchanged — see [§5](#5-the-materialized-view-problem).

## 5. The materialized-view problem

ClickHouse incremental MVs are **INSERT triggers**. They fire only on inserts into `events`; an
`ALTER TABLE events DELETE` mutation does **not** fire them and does **not** roll back the aggregate
state they already wrote. (The migrations confirm this: both rollups carry a one-time backfill
commented *"The MV only captures inserts after it is created."*) So **every derived store must be
erased explicitly.**

| Store (migration) | Trigger type | Keyed by | Holds person's PII after event delete? | Action |
| --- | --- | --- | --- | --- |
| `events` (001) | base table | — | yes (raw rows) | `ALTER … DELETE WHERE distinct_id IN keys` |
| `distinct_id_activity_states` (005) | incremental | `(project_id, distinct_id)` | **yes** — first/last seen, sessions, latest browser/OS/device/geo | `ALTER … DELETE WHERE distinct_id IN keys` |
| `dashboard_session_rollup` (007) | incremental | `(project_id, kind, session_id)` | **yes** — entry/exit URL + geo + UTM per session | `ALTER … DELETE WHERE session_id IN session_ids` |
| `profile_aliases` (002) | base table | `(project_id, profile_id, alias_id)` | yes — links | `ALTER … DELETE WHERE profile_id = ?` |
| `profiles` (003) | base table | `(project_id, id)` | yes — `properties` (email/name) | `ALTER … DELETE WHERE id IN keys` — every frozen distinct_id, so merge-tombstone rows keyed by absorbed anon ids (and seeded anon rows whose id = distinct_id) are physically removed too |
| `dashboard_event_rollup_daily` (006) | incremental | `(project_id, kind, dim_name, day, dim_value)` | only as `cnt` + `uniqState` HLL — **no per-person key** | **left** (decision "a"; anonymous aggregate) |
| `event_names` (004) | incremental | `(project_id, kind)` | no | left — not personal data |
| `property_keys_event_buckets` (004) | refreshable APPEND | property key/type buckets | no (key *names*/types, never values) | left — schema metadata |
| `property_keys_profile_current` (004) | refreshable rebuild | from `profiles FINAL WHERE is_deleted=0` | no | **self-heals** on next 5-min refresh |

### Why the `events` sort key is NOT reordered

The roadmap flags that `distinct_id` isn't in the sort key
(`ORDER BY (project_id, toStartOfMinute(occur_time), kind, event_id)`). Reordering `distinct_id`
ahead of time would wreck the time-range scans every insights query depends on, to optimize a *rare,
async* delete. The existing **`idx_distinct_id` bloom-filter skip index** already lets the mutation
prune parts. Keep migration 001 as-is.

### Critical sequencing — freeze before delete

`session_ids` are resolved **from `events`**, so they must be captured **before** the events are
deleted, then **frozen** onto the `compliance_requests` row. Otherwise a retry after the events delete
re-queries empty and silently leaves `dashboard_session_rollup` rows behind. Freezing both
`distinct_ids` and `session_ids` on the first pass makes the whole worker idempotent under redelivery.

## 6. Erasure flow

### Synchronous (RPC → service, one PostgreSQL transaction)

1. Resolve the subject — by `id` (`Delete`) or `external_id` (`DeleteDataSubject`). A profile may
   **not** exist — allowed on the external_id path, and on the by-id path when the id resolves to a
   derived anonymous person / alias / external_id via the resolution ladder
   ([§2](#derived-anonymous-persons-are-erasable-by-id)).
2. In a PG tx: insert `compliance_requests` (kind `erase`, `pending`, with `external_id`, `profile_id` if any,
   `requested_by`); **if a profile exists**, soft-delete it (`deletion_time = now()`) and deactivate
   its devices (immediate read-hiding). Commit.
3. Publish `compliance.erase` → `{ project_id, request_id, profile_id?, external_id }`.
4. Return `{ request_id, status: pending }`.

Soft-delete is now just **phase 1** of erasure (hide immediately), not a terminal state.

### Asynchronous (the `compliance` worker — `internal/app/workers/compliance/`)

Consumer `compliance-erase-processor` on a dedicated `compliance` stream, subject `compliance.erase`,
DLQ → `dlq.compliance.erase`. The worker is one process hosting one consumer per compliance job;
4.2 export and 4.5 retention add sibling consumers alongside erase (see decision 6).

1. Load the request; if already `completed` → ack (idempotent no-op). Else mark `processing`.
2. **Freeze** (if not already on the row): compute `distinct_id_keys` + `session_ids`
   ([§2](#2-identity-model-what-to-erase)); persist both to `compliance_requests`.
3. **PostgreSQL hard deletes** (only if `profile_id` present), in this order:
   - `DELETE FROM profile_devices WHERE profile_id = ? AND project_id = ?`
   - `DELETE FROM profiles WHERE id = ? AND project_id = ?`

   > **Order is load-bearing:** `profile_devices.profile_id` is `references profiles(id) **on delete
   > set null**` (`007_create_profile_devices.sql:11`). Deleting `profiles` first would only *null*
   > the link and orphan the device rows — leaving the push token + endpoint (a delivery secret)
   > behind, findable only by token. Devices must be hard-deleted first, by `profile_id`.
4. **ClickHouse mutations** (`mutations_sync = 1`), each idempotent:
   - `ALTER TABLE dashboard_session_rollup DELETE WHERE project_id = ? AND session_id IN (…)`
   - `ALTER TABLE events                   DELETE WHERE project_id = ? AND distinct_id IN (…)`
   - `ALTER TABLE distinct_id_activity_states DELETE WHERE project_id = ? AND distinct_id IN (…)`
   - `ALTER TABLE profile_aliases          DELETE WHERE project_id = ? AND profile_id = ?`   (if profile)
   - `ALTER TABLE profiles                 DELETE WHERE project_id = ? AND id IN (…)`         (every frozen distinct_id — covers merge tombstones + anon-keyed rows)
5. Mark `completed`, set `completed_at`, record `events_identified`. Ack.
6. **Errors:** decode/validate failures → `PermanentError` → DLQ. Transient CH/PG errors → Nak →
   retry (safe: frozen lists keep step 2 stable even after events are gone). On the **final**
   delivery the worker marks the row `failed` (so the ledger never reports a stalled request as
   still `processing`); a controller re-request then re-drives the existing row — reviving a
   `failed` one — via `reopenErasure` rather than duplicating it. NATS redelivery (up to
   `max_deliver`) plus that re-request path are the durability backstops; there is **no** automated
   reconciliation sweep (see §9 for the resulting SLA gap and the `ListStuckComplianceRequests`
   alerting backstop).

## 7. ClickHouse mutation strategy

The NATS worker framework has **no ack-heartbeat** — each message gets a fixed `ProcessingTimeout`
(`worker.go:270` wraps it in `context.WithTimeout`) and a single `AckWait`; `InProgress()` is only a
test stub, never called in the consume loop. So the worker cannot hold a message open indefinitely
while a slow mutation runs.

- **Strategy A (chosen, decision 5):** give the erase worker a generous `ProcessingTimeout`/`AckWait`
  (~5 min) and block on `mutations_sync = 1`. Simple and self-healing — a mutation that exceeds the
  window just redelivers and re-runs idempotently. Appropriate at current scale.
- **Strategy B (scale path, deferred):** issue `mutations_sync = 0`, mark `processing`, and confirm
  via a `system.mutations` poll / sweeper before flipping to `completed`. Decouples from `AckWait`;
  more moving parts. Revisit if single-subject deletes routinely exceed a few minutes.

## 8. Tests & acceptance

- **Integration — full erase:** alias an `anon-<uuid>` into an `external_id`; ingest events under
  *both*; ingest a session. Run erasure. Assert: events for **all** `distinct_id` keys gone;
  `distinct_id_activity_states` gone; `dashboard_session_rollup` rows for those sessions gone; PG
  `profiles` + `profile_devices` and CH `profiles` + `profile_aliases` gone; `compliance_requests` =
  `completed` with counts; re-query empty. Assert `dashboard_event_rollup_daily` **unchanged** (pins
  decision "a").
- **Integration — no-profile external_id:** ingest events keyed only by a raw `external_id` (no
  profile). Erase by `external_id`. Assert those events + their activity/session rollups are gone and
  the request completes.
- **Integration — device ordering:** assert no orphaned `profile_devices` row (nulled `profile_id`)
  survives an erasure.
- **Retry / idempotency:** run the worker twice, and again after a simulated crash *after* the events
  delete — the session rollup is still cleaned via the frozen `session_ids`; the request stays
  `completed`.
- **Unit:** the fan-out resolver; rejection of public-SDK-key callers on the erasure RPCs; the
  no-profile path.

## 9. Known limitations (deferred)

- **`dashboard_event_rollup_daily`** retains the subject's contribution inside `cnt` (sum) and
  `uniqState(distinct_id)` (a non-reversible HLL sketch). No per-person key exists to target. Treated
  as an anonymous aggregate (GDPR Recital 26). Exact removal would require rebuilding the affected
  `(project_id, day)` ranges from `events FINAL` — revisit only if a granular `dim_value` (e.g. city)
  with `cnt = 1` becomes a re-identification concern.
- **Multiple historical profiles for one `external_id`:** the `(project_id, external_id)` unique index
  covers only *active* profiles, so older soft-deleted rows with the same `external_id` may carry
  their own anon aliases that the active-profile resolution misses. v1.1: resolve *all* profile rows
  (active + soft-deleted) for the `external_id` and union their alias sets.
- **`property_keys_event_buckets`** retains property **key names + types** (never values, never ids)
  in already-written buckets until normal partition age-out.
- **No automated re-drive for a lost or aged-out enqueue (SLA gap).** After the prelude commits
  (audit row + soft-delete), if `publishErase` fails — or the worker is down past the `compliance`
  stream's `max_age` (720h) so JetStream drops the message — the row sits `pending`/`processing` with
  nothing to re-publish it. The subject *looks* erased (profile hidden) while PII is physically intact,
  and recovery is **operator-driven**: re-submit the subject via `RequestErasure*` (frozen identifiers
  keep the re-drive correct) — there is no reaper. The `ListStuckComplianceRequests(olderThan, limit)`
  service method (open rows older than a threshold) is the **alerting backstop** so oncall can spot and
  re-drive these; wire it to monitoring. A periodic auto-sweeper that re-publishes stuck rows is the
  natural fast-follow (deliberately deferred here to keep destructive re-drives operator-initiated and
  auditable).
