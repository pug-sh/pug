# 4.2 — Export / access that reaches the events: implementation scope

Scopes **§4.2** of [`roadmap.md`](./roadmap.md) — the data-subject **access + portability**
right (GDPR Art. 15 / Art. 20, DPDP §11). Today no endpoint aggregates a subject's data:
`ProfilesService.Get/GetByExternalId/List` return profile rows only, never events. A controller
who receives a "give me my data" request from one of their end-users has no mechanism to fulfil it.

**Status:** design agreed, not yet implemented.
**Context:** pre-launch — **no backward-compatibility constraints** (proto/schema may change freely).
Build for the stricter GDPR/DPDP superset. Sibling to [`4.1-erasure-scope.md`](./4.1-erasure-scope.md);
export **reuses 4.1's identity fan-out** and the same shared/private auth boundary.

---

## Decisions locked during scoping

1. **Synchronous streaming delivery + a lightweight audit row** (chosen over an async artifact job).
   `ExportDataSubject` server-streams the subject (profile + paginated events) directly in the RPC
   response, reusing `events.Reader.GetActivityFeed`. There is **no worker job, no artifact store, and
   no new infrastructure.** The controller's backend consumes the stream and packages/delivers the
   file to its end-user. This matches the roadmap's *"one call returns everything we hold about a
   subject, paginated for large histories"* and — unlike a stored artifact — does **not** create a
   second durable copy of personal data that would itself need retention + erasure reach.
2. **This revises 4.1's decision 6.** The compliance worker was provisioned to host export as a
   sibling consumer; export turns out not to need an async job, so it stays out of the worker. The
   worker remains the home for erasure (§4.1) and the genuinely-async §4.5 retention/TTL purge.
3. **Reuse the existing read path verbatim.** `events.Reader.resolveProfileIDs`
   (`internal/core/events/reader.go:379`) already resolves the full `distinct_id` fan-out
   (profile id + external_id + alias_ids), and `GetActivityFeed` (`reader.go:630`) already returns
   full event rows keyset-paginated on `(occur_time DESC, event_id DESC)`. Export adds **no** new
   ClickHouse query. (`internal/core/events` does not import `internal/core/profiles`, so the
   `profiles → events` reuse is cycle-safe.)
4. **Export reuses the unified `compliance_requests` table** (`015_create_compliance_requests.sql`)
   with `kind = 'export'` — **no new table**. The 4.1 `deletion_requests` table was renamed to
   `compliance_requests` and given a `kind` column in place (pre-launch, was not yet committed) so
   erasure and export share **one DSAR ledger** — exactly the single audit log §5 of the roadmap
   calls for. The erase-only `distinct_ids`/`session_ids` columns sit NULL for export rows; the count
   lives in the shared `events_affected` column (surfaced as `events_exported` on the export reader).
5. **Server-streaming, not unary-paginated.** A `stream` response gives a deterministic end-of-data
   signal, so the audit row can be **finalized exactly once** at stream completion (`completed` +
   count, or `failed` on mid-stream error/disconnect). A unary `(…, page_token)` loop can't tell the
   server when the client is "done", so the audit trail would be unreliable. Mirrors the existing
   `List` server-stream idiom.
6. **Shared service only (private API key or JWT), never the public SDK.** Identical reasoning to
   erasure, and the stakes are higher: an export endpoint reachable with an extractable public key is
   a **bulk-exfiltration** vector (read every subject's full history), strictly worse than the
   erasure-DoS it mirrors. Export must be driven by the controller's verified backend.
7. **Object storage is the documented future scale path.** We may later need to write the export as a
   durable artifact to object storage (S3/MinIO/GCS) and hand back a short-lived signed URL — for
   very large histories, or to let a controller re-download / disintermediate delivery. That is
   deliberately **out of scope now** (no object storage exists in the stack today) but the design
   below is structured so it slots in without reshaping the API — see [§7](#7-future-object-storage-artifacts).

## 1. Goal & acceptance

**Done when:** one authenticated call returns **everything pug holds about a subject** —
- the profile row + traits (`properties`) and its activity summary, when a profile exists;
- **every event row across all of the subject's `distinct_id`s** (the §4.1 fan-out: external_id +
  profile id + all alias_ids), with full `auto_properties` + `custom_properties`;
- the subject's device registrations (metadata only — the raw push token is redacted, see
  [§2](#2-identity-model-what-to-export));
- streamed in pages so an arbitrarily large history never buffers in memory;
- a `compliance_requests` row (kind `export`) recording **who** exported **what** and **when**, with
  the event count, marked `completed` — so the controller can later *prove* the access/portability
  request was fulfilled (the §5 DSAR audit-log duty).

Export **never 404s on an absent subject**: an unknown `external_id` yields a valid empty export
(empty subject, zero events, audit row with `events_exported = 0`) — "we hold nothing about this
person" is a legitimate DSAR answer, not an error.

## 2. Identity model: what to export

Export resolves the **same identity fan-out as erasure** — so "what we export" and "what we would
erase" are provably the same set. The read path already encapsulates it:

```
distinct_id_keys = resolveProfileIDs(project_id, <external_id | profile_id>)   -- reader.go:379
                 = {profile.id} ∪ {current external_id} ∪ {all current alias_ids}
                 -- when nothing resolves (no profile), returns the input key unchanged,
                 -- so events keyed directly by a raw external_id are still exported.
```

This is the live resolver (not a frozen set — export is a read, nothing to make retry-safe).

### Data inventory — what goes in the export

| Source | Field(s) | In export? |
| --- | --- | --- |
| CH `profiles` (via existing read) | external_id, profile_id, create_time, **traits** (`properties`), activity summary | ✅ the `subject` chunk |
| CH `events` (all distinct_ids) | event_id, distinct_id, kind, occur_time, session_id, **auto_properties** (geo, device, UA, url, utm), **custom_properties** | ✅ the `events` chunks |
| PG `profile_devices` | platform, created_at, active | ✅ metadata only |
| PG `profile_devices.token` | raw push token + endpoint | ❌ **redacted** — a delivery *secret*, not portable personal data; disclosing it adds breach risk with no portability value (decision flagged) |
| CH rollups (`*_activity_states`, `dashboard_*_rollup`) | aggregates | ❌ derived, not source data; the raw events they summarize are already exported |

### `external_id` with no profile

Symmetric to erasure: `ExportDataSubject(external_id)` proceeds even when no profile row resolves.
The `subject` chunk carries an empty `profile_id` + empty traits; the event stream still returns every
row keyed by that raw `external_id`. The only thing unavailable without a profile is the anon-id
history (nothing links those anon ids to the person) — the same inherent, accepted limit as §4.1.

## 3. API surface

All export RPCs live on the **shared** service (`proto/shared/profiles/v1/profiles.proto`) — **private
API key or JWT**, never the public SDK (decision 6).

```proto
service ProfilesService {
  // ... existing Get / GetByExternalId / List / Delete / DeleteDataSubject / GetDeletionRequest ...

  // Export streams everything pug holds about a data subject identified by
  // profile id (the dashboard handle).
  rpc Export(ExportRequest) returns (stream ExportResponse) {}
  // ExportDataSubject streams everything pug holds about a data subject
  // identified by external_id (the controller-facing DSAR handle). Proceeds even
  // when no profile resolves. See docs/compliance/4.2-export-scope.md.
  rpc ExportDataSubject(ExportDataSubjectRequest) returns (stream ExportResponse) {}
  // GetExportRequest returns the audit row for an export so a controller can
  // prove an access/portability request was fulfilled (the DSAR audit trail).
  rpc GetExportRequest(GetExportRequestRequest) returns (GetExportRequestResponse) {}
}

message ExportRequest           { string id = 1 [(buf.validate.field).required = true]; }
message ExportDataSubjectRequest { string external_id = 1 [(buf.validate.field).required = true]; }

// Streamed in order: exactly one `subject` chunk first (carries the request_id +
// traits + devices), then zero or more `events` chunks (one per page).
message ExportResponse {
  oneof chunk {
    ExportSubject subject = 1;
    ExportEvents  events  = 2;
  }
}

message ExportSubject {
  string request_id = 1;                       // audit handle → GetExportRequest
  string external_id = 2;
  string profile_id = 3;                       // empty when exporting by external_id with no profile
  google.protobuf.Struct traits = 4;           // profiles.properties; empty when no profile
  google.protobuf.Timestamp created_at = 5;
  ProfileActivitySummary activity = 6;         // reuse the existing message
  repeated ExportDevice devices = 7;           // push subscriptions; token redacted
}

message ExportDevice {
  string platform = 1;
  google.protobuf.Timestamp created_at = 2;
  bool active = 3;
  // raw push token + endpoint deliberately omitted (delivery secret).
}

message ExportEvents { repeated ExportEvent events = 1; }

message ExportEvent {
  string event_id = 1;
  string distinct_id = 2;
  string kind = 3;
  google.protobuf.Timestamp occur_time = 4;
  string session_id = 5;
  google.protobuf.Struct auto_properties = 6;
  google.protobuf.Struct custom_properties = 7;
}

message GetExportRequestRequest { string request_id = 1 [(buf.validate.field).required = true]; }
message GetExportRequestResponse {
  string request_id = 1;
  string external_id = 2;
  string profile_id = 3;
  string status = 4;                           // pending | completed | failed
  int64 events_exported = 5;
  google.protobuf.Timestamp requested_at = 6;
  google.protobuf.Timestamp completed_at = 7;
}
```

- `Export` (by id) and `ExportDataSubject` (by external_id) are two front doors to **one** streaming
  pipeline, mirroring `Delete` / `DeleteDataSubject`. Both pass their key straight into
  `resolveProfileIDs`, which accepts a profile id, alias id, or external_id transparently.
- **Format:** the proto stream *is* the structured, machine-readable artifact (GDPR Art. 20). Connect's
  JSON codec serves it as JSON-over-HTTP for free when the controller calls with
  `Content-Type: application/json` — no separate serializer. CSV is **not** produced server-side
  (nested `properties` don't flatten losslessly); a controller that wants CSV flattens the JSON
  client-side. See [§6](#6-known-limitations-deferred).

## 4. Schema changes

### No new table — reuse `compliance_requests` with `kind = 'export'`

Export records its audit row in the unified `compliance_requests` ledger (decision 4); there is **no
`016` migration**. It reuses the existing `CreateComplianceRequest` (passing `kind = 'export'`) and
adds two small write queries to `queries/write/compliance_requests.sql`:

- `FinalizeComplianceRequestExport(id, project_id, events_affected)` → sets `events_affected`,
  `status = 'completed'`, `completed_at = now()`. Export has no freeze stage, so the count is written
  at completion (unlike erase, which freezes it on the first worker pass).
- reuses `MarkComplianceRequestFailed(id, project_id, error)` on mid-stream failure.

`status = 'processing'` is unused for export (delivery is synchronous; there is no worker stage) — the
row goes `pending → completed | failed` within one handler. `profile_id` is not a foreign key (it must
outlive a later erasure of the subject) — that holds for every `compliance_requests` row.

### No ClickHouse or NATS change

Export is a pure read over existing tables. No CH migration, no new stream/consumer/subject, no CLI
or deploy wiring.

## 5. Export flow (synchronous, in the RPC handler)

The whole operation lives in the shared `ProfilesService` handler + a `internal/core/profiles/export.go`
method; the handler owns `stream.Send`, the core method owns resolution, paging, and the audit row.

1. **Auth + resolve.** `MustGetPrincipalWithProject`. The subject key is the `id` (`Export`) or
   `external_id` (`ExportDataSubject`). A profile may not exist (external_id path) — allowed.
2. **Open audit row.** `CreateComplianceRequest` (`kind = 'export'`, `pending`, `external_id`,
   `profile_id` if resolved, `requested_by`). `request_id = xid.New()`.
3. **Emit the `subject` chunk.** Read the profile via the existing
   `coreprofiles.GetByExternalID`/`GetByID` (empty on `ErrProfileNotFound`); read devices from
   `profile_devices` (token redacted). Send `ExportSubject{request_id, external_id, profile_id,
   traits, created_at, activity, devices}`.
4. **Stream events.** Loop `events.Reader.GetActivityFeed{ProjectID, DistinctID: <key>,
   PageSize: MaxPageSize, PageToken: cursor}` → send one `ExportEvents` chunk per page, accumulate
   the count, advance the keyset cursor until it returns nil. Memory stays bounded at one page
   (≤1000 events).
5. **Finalize once.** On clean completion: `FinalizeComplianceRequestExport(events_affected = count)`
   → `status = completed`, `completed_at = now()`. On any mid-stream error **or client disconnect**
   (`ctx.Err()`): `MarkComplianceRequestFailed(error)` → `status = failed`. The partial bytes already
   sent are the client's to discard; the audit row tells the truth.

### Why the audit row is written even though delivery is synchronous

§5 of the roadmap requires we can show *"what was deleted/exported, when."* Both erasure and export
record it in the shared `compliance_requests` ledger (by `kind`). Without the row, a synchronous
export would leave no trace that an Art. 15/20 request was serviced.

## 6. Known limitations (deferred)

- **Duplicate event deliveries may appear.** `GetActivityFeed` reads without `FINAL` (relies on
  background-merge dedup), so a redelivered event can surface twice — the same bounded inaccuracy
  documented for the rollup fast-path and activity feed. Acceptable for an export; a controller can
  dedup on `event_id`. Adding `FINAL` to a full-history scan is too costly to justify here.
- **Multiple historical profiles for one `external_id`.** Same caveat as erasure's limitation #2: the
  live resolver only follows the *active* profile's aliases, so anon ids belonging to older
  soft-deleted profiles with the same `external_id` may be missed. v1.1: union all profile rows
  (active + soft-deleted) for the `external_id`.
- **Push token redacted.** Device export carries platform/created_at/active but not the raw token +
  endpoint (delivery secret). If a regulator ever requires the literal token, revisit.
- **No durable artifact / no re-download.** By design (decision 1) — the controller fetches once and
  packages delivery itself. The object-storage path ([§7](#7-future-object-storage-artifacts))
  covers the case where pug must retain a downloadable artifact.
- **CSV not produced server-side.** Proto/JSON only; nested `properties` don't flatten losslessly.

## 7. Future: object-storage artifacts

We may later need export to produce a **durable file** in object storage (S3/MinIO/GCS) and return a
short-lived **signed URL** — to handle very large histories without holding a stream open, to let a
controller re-download, or to deliver to the end-user directly. None of that infrastructure exists in
the stack today, so it is out of scope now, but the current design is deliberately forward-compatible:

- **Reuse the audit row as the job row.** `compliance_requests` gains `artifact_location` +
  `artifact_expires_at` columns (additive); export rows begin using the existing `processing` status.
  It becomes the async job/status record — still no new table.
- **Add the worker consumer that was provisioned in 4.1.** `RequestDataSubjectExport` enqueues
  `compliance.export`; the compliance worker resolves the same fan-out, streams events to a
  **gzipped NDJSON** writer, uploads to the bucket, and flips the row to `completed` with the
  artifact location. `GetExportRequest` then returns a signed URL. This is exactly the sibling
  consumer reserved in `internal/app/workers/compliance/compliance.go`.
- **The synchronous `ExportDataSubject` stream can stay** as the low-latency path for small subjects,
  with the async artifact path for large ones — same core serializer feeding both.
- **New obligation:** the artifact is itself a fresh copy of personal data, so it must carry its own
  TTL (`artifact_expires_at` + a purge) and fall within §4.1 erasure + §4.5 retention reach. Object
  storage here also dovetails with the S3 backups flagged as open in the infra repo (roadmap §4.8).

## 8. Tests & acceptance

- **Integration — full export:** alias an `anon-<uuid>` into an `external_id`; ingest events under
  *both* + a session; set profile traits + a device. Export by `external_id`. Assert: subject chunk
  carries traits + activity + device (no token); **every** event across all `distinct_id` keys is
  streamed with `auto`/`custom` properties intact; the count matches; `export_requests` =
  `completed` with `events_exported = N`.
- **Integration — no-profile external_id:** ingest events keyed only by a raw `external_id` (no
  profile). Export by `external_id`. Assert: empty `profile_id`/traits, all events streamed, request
  `completed`.
- **Integration — empty subject:** unknown `external_id` → empty subject chunk, zero event chunks, no
  error, audit row `completed` with `events_exported = 0`.
- **Integration — large history / pagination:** ingest > `MaxPageSize` events; assert all stream
  across multiple `ExportEvents` chunks (keyset cursor correctness, no gaps/dupes beyond the
  documented redelivery case).
- **Integration — audit proof:** after an export, `GetExportRequest(request_id)` returns the row with
  the right count + timestamps.
- **Unit:** chunk ordering (exactly one `subject` first, then `events`); finalize-completed vs
  fail-on-ctx-cancel; rejection of public-SDK-key callers on the export RPCs.
